1 Visão geral

Quando um usuário com papel de owner ou admin acessa /dashboard, o sistema renderiza o AdminDashboard — uma página que consolida em tempo real a operação da empresa no mês corrente.

4 KPIs principais

  • OS abertas
  • OS entregues no mês
  • Faturamento do mês
  • Despesas operacionais

3 painéis de detalhe

  • Comparativo entre filiais
  • Resumo financeiro consolidado
  • Últimas 6 ordens de serviço

Stack envolvida

  • Next.js 16 (Server Component)
  • Server Action para buscar dados
  • Supabase como banco
  • RLS + companyId para isolar dados

2 Arquitetura e fluxo

Tudo acontece em uma única renderização no servidor. Sem loading spinners no cliente — os dados chegam prontos quando a página é entregue.

Usuário acessa a rota
2. Proxy

3. Server Component
page.tsx
Escolhe admin ou técnico
4. Server Action
getDashboardOverview
Busca dados em paralelo
5. Supabase
6 queries paralelas
Promise.all nas tabelas
6. Agregação
Cálculo no Node
Soma, agrupa por filial
7. HTML
Renderização
Página pronta para o navegador
Por que Server Action? Os dados são sensíveis e o cálculo exige acesso a várias tabelas. Resolver no servidor reduz chamadas do cliente, evita expor consultas e aproveita proximidade com o banco.

3 Período de apuração

O dashboard sempre olha para o mês corrente, do dia 1 até hoje. É calculado a cada requisição. 

const now = new Date()
const start = new Date(now.getFullYear(), now.getMonth(), 1)
// startDate = "2026-04-01"
// endDate   = "2026-04-19"
// label     = "01/04 a 19/04"
  • Início: primeiro dia do mês atual, 00:00
  • Fim: hoje, 23:59:59.999
  • Fuso: servidor (UTC); exibição formatada em PT-BR
  • O rótulo do período é exibido no topo do dashboard para deixar claro a janela considerada

OS abertas

Conta todas as ordens de serviço com status que indica trabalho em andamento, sem filtro de data. Os status considerados "abertos" são:

aguardando em_analise aguardando_aprovacao aprovado reprovado aguardando_peca enviado_terceiro pronto

OS entregues

Conta OS com status finalizado cujo delivered_at está dentro do mês corrente. Ou seja, a data que conta é a de entrega ao cliente, não a de abertura.

Faturamento

Soma do campo net_amount de todos os lançamentos em cash_entries no mês. Representa o que efetivamente entrou em caixa.

O ticket médio mostrado no cartão é faturamento / número de OS pagas, considerando OS distintas (uma OS pode ter várias entradas de caixa).

Despesas operacionais

Soma do amount das contas a pagar (bills) com due_date dentro do mês. Atenção: usa a data de vencimento, não a data de pagamento.

Resultado (net): exibido embaixo do cartão de despesas e também na seção "Resumo financeiro". É simplesmente faturamento − despesas.

6 Comparativo entre filiais

Cada filial ativa aparece como uma linha. Os valores são agregados no Node.js após as queries, usando Map para contagem por branch_id.

Coluna Origem Cálculo
OS abertas service_orders (abertas) Contagem agrupada por branch_id
OS entregues service_orders (entregues no mês) Contagem agrupada por branch_id
Faturamento cash_entries Soma de net_amount por filial
Despesas bills Soma de amount por filial
Resultado — faturamento − despesas

A ordenação é por faturamento decrescente, com desempate pelo número de OS abertas. A barra verde no nome da filial mostra proporcionalmente a receita em relação à maior do grupo.

7 Resumo financeiro

Painel simples no canto direito do dashboard, consolidando os três números principais do mês.

  • Faturamento — positivo soma dos cash_entries.net_amount
  • Despesas — negativo soma dos bills.amount
  • Resultado — positivo ou negativo conforme o sinal da diferença

É o mesmo dado mostrado nos KPIs do topo, só que reorganizado em formato de "demonstrativo".

8 Últimas ordens de serviço

Mostra as 6 OS mais recentes (ordenadas pelo número decrescente), independente do status.

Coluna exibida Origem

Equipamento concatenação de device_type, device_brand e device_model

Cada número de OS é um link para a tela de detalhes /dashboard/ordens-de-servico/[id].

9 Segurança e isolamento

O dashboard aplica três camadas de proteção antes de devolver qualquer dado:

1

Proxy (camada de rota)

Verifica sessão Supabase, bloqueia usuários inativos e redireciona não autenticados para /login.

2

Server Component (camada de role)

A página page.tsx escolhe AdminDashboard ou TechnicianDashboard com base em app_metadata.role. Atendentes e técnicos caem no dashboard operacional.

3

Server Action (camada de tenant)

getAdminContext('financeiro') valida permissão e extrai o companyId do usuário. Todas as queries aplicam WHERE company_id = ? no banco.

Multi-tenant: o sistema é multi-empresa. O isolamento dos dados de cada cliente é garantido pelo company_id em todas as consultas + políticas RLS no Supabase.

10 Limitações atuais e próximos passos

Limitações

  • Período fixo: sempre o mês corrente, sem seletor para comparar com meses anteriores
  • "Despesas" considera vencimento, não pagamento efetivo — pode distorcer o resultado se houver muitas contas em atraso
  • Limite de 5.000 registros por query — empresas muito grandes podem precisar de paginação ou pré-agregação
  • Não há cache: a cada recarregamento da página, as 6 queries são executadas de novo
  • Sem gráficos de evolução temporal (só números consolidados)

Possíveis evoluções

  • Seletor de período (hoje, semana, mês, trimestre)
  • Gráficos de evolução diária/semanal de OS abertas e faturamento
  • Comparação com o mês anterior (delta %)
  • Cache em tempo curto para reduzir carga no banco
  • Materialização em view ou tabela de agregados
